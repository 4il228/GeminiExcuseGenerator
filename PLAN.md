# PLAN.md: Атомарный план разработки «Генератора отмазок»

---

## 👨‍💻 Общая матрица ролей и контекста

Чтобы исключить перегрузку контекстного окна ИИ-разработчиков, каждый агент получает изолированное задание, входные артефакты и проверяемые критерии приемки. Агентам **запрещено** изучать всю кодовую базу — они оперируют только выделенной им областью.

* **Agent Backend (A-BE):** Специализируется на Python 3.13, FastAPI, валидации Pydantic и интеграции Google Gemini API.
* **Agent Frontend (A-FE):** Специализируется на современной верстке (HTML/Tailwind CSS), архитектуре Single Page Application (SPA) и взаимодействии с API (Fetch/Axios).
* **Agent DevOps/QA (A-DQ):** Специализируется на контейнеризации (Docker), управлении секретами (`.env`) и автоматизированном тестировании стабильности контрактов данных.

---

## 🛠 Фаза 1: Инициализация окружения и Секьюрити-базис (Роль: A-DQ)

### Шаг 1.1: Создание структуры репозитория и конфигурации зависимостей

* **Описание задачи:** Создать скелет монорепозитория, настроить конфигурационные файлы для изоляции бэкенда и фронтенда. Подготовить менеджер пакетов для Python 3.13.
* **Входной контекст:** Отсутствует.
* **Действие:**
1. Инициализировать следующую файловую структуру:
```
├── backend/
│   ├── app/
│   │   ├── __init__.py
│   │   └── main.py
│   ├── requirements.txt
│   └── .env.example
├── frontend/
│   ├── index.html
│   └── styles.css
└── README.md

```


2. В `backend/requirements.txt` зафиксировать версии: `fastapi`, `uvicorn`, `pydantic`, `google-generativeai`.


* **Критерии приемки:** Команда `pip install -r backend/requirements.txt` успешно выполняется на Python 3.13. Структура директорий строго соответствует схеме.

### Шаг 1.2: Создание конфигурации переменных окружения

* **Описание задачи:** Подготовить файл `.env.example` со всеми необходимыми ключами для безопасного развертывания проекта.
* **Входной контекст:** Требования из `SPEC.md` по переменным `ENV`, `HOST`, `PORT`, `GEMINI_API_KEY`.
* **Действие:** Создать файл `backend/.env.example` с заполнителями для ключей и дефолтными сетевыми настройками.
* **Критерии приемки:** Файл `.env.example` присутствует, оригинальный `.env` добавлен в глобальный `.gitignore`.

## ⚙️ Фаза 2: Разработка бэкенда и валидации (Роль: A-BE)

### Шаг 2.1: Проектирование схемы данных и эндпоинта-заглушки (Mock)

* **Описание задачи:** Создать схемы валидации входных и выходных данных, а также зарегистрировать базовый эндпоинт, отдающий мок-данные.
* **Входной контекст:** Описание Pydantic-моделей из секции 3 в `SPEC.md`.
* **Действие:**
1. Реализовать `SituationEnum` (значения: `late_work`, `missed_homework`, `forgot_birthday`, `dog_ate_code`).
2. Реализовать Pydantic-модели `ExcuseRequest` и `ExcuseResponse`.
3. Создать в `backend/app/main.py` маршрут `POST /api/v1/generate`, возвращающий захардкоженный JSON, соответствующий схеме.


* **Критерии приемки:** Сервер успешно запускается через `uvicorn app.main:app`. Запрос на `/api/v1/generate` с валидным `situation_id` возвращает статус 200 и мок-данные. Запрос с невалидным ID возвращает статус 422.

### Шаг 2.2: Интеграция с Google Gemini API через Structured Outputs

* **Описание задачи:** Подключить официальный SDK `google-generativeai`, настроить безопасную сборку системного промпта по словарю-маппингу и применить `response_schema` для детерминированного ответа.
* **Входной контекст:** Реализованный Шаг 2.1; словарь `SITUATION_MAP` и конфигурация `GenerateContentConfig` из `SPEC.md`. Переменная `GEMINI_API_KEY` берется из окружения.
* **Действие:**
1. Интегрировать инициализацию `genai.configure`.
2. Заменить мок-логику эндпоинта на реальный вызов `model.generate_content()`.
3. Передать в конфигурацию генерации `response_schema=ExcuseResponse` и `temperature=0.7`.


* **Критерии приемки:** Код бэкенда не содержит конкатенаций пользовательских строк внутри промпта. Вызов LLM возвращает строго валидный JSON, который успешно парсится в `ExcuseResponse`.

## 🎨 Фаза 3: Разработка интерфейса и верстка (Роль: A-FE)

### Шаг 3.1: Верстка каркаса и селектора ситуаций (Tailwind CSS)

* **Описание задачи:** Создать адаптивный одностраничный интерфейс, содержащий интерактивные элементы выбора ситуации.
* **Входной контекст:** Гайдлайн по стилям и разметка компонентов из секции 4 в `SPEC.md`. Контекст бэкенда отсутствует.
* **Действие:**
1. В `frontend/index.html` подключить Tailwind CSS через CDN.
2. Сверстать сетку карточек (Grid Layout) для ситуаций из `SituationEnum`. Добавить им hover-эффекты и состояния фокуса.
3. Каждой карточке добавить `data-situation-id` атрибут, соответствующий значениям бэкенда.


* **Критерии приемки:** Интерфейс корректно отображается как на мобильных устройствах, так и на десктопе. Клик по карточке визуально выделяет её (добавить класс активного состояния).

### Шаг 3.2: Верстка блока выдачи отмазки и логики копирования

* **Описание задачи:** Сверстать блок отображения результатов с динамическим индикатором правдоподобности и кнопкой копирования текста.
* **Входной контекст:** Спецификация компонента `Excuse Output Slot` из `SPEC.md`.
* **Действие:**
1. Интегрировать скрытый по умолчанию блок результата под селектором ситуаций.
2. Сверстать элемент рейтинга правдоподобности (badge с цветным индикатором).
3. Написать изолированный JS-скрипт внизу страницы, который вешает обработчик на кнопку «Скопировать отмазку» и сохраняет текст в `navigator.clipboard`.


* **Критерии приемки:** Блок визуально соответствует спецификации. При ручном удалении класса `hidden` блок корректно отображается. Нажатие на кнопку копирования успешно сохраняет демонстрационный текст в буфер обмена.

## 🔌 Фаза 4: Интеграция слоев и сквозное тестирование (Роли: A-FE + A-DQ)

### Шаг 4.1: Реализация API-клиента на фронтенде (Роль: A-FE)

* **Описание задачи:** Написать JavaScript-код для отправки выбранного ID на бэкенд и динамического обновления интерфейса результатами ответа.
* **Входной контекст:** Готовая верстка из Фазы 3, контракт данных `POST /api/v1/generate` из Фазы 2.
* **Действие:**
1. Написать функцию `async function generateExcuse(situationId)`.
2. Реализовать отправку запроса через `fetch()` на `/api/v1/generate`.
3. Добавить обработку состояния загрузки (лоадер вместо текста отмазки во время ожидания ответа).
4. При успешном ответе: удалить класс `hidden` у блока выдачи, вставить текст в тег отмазки, динамически изменить цвет индикатора в зависимости от значения `believability_score`.


* **Критерии приемки:** При клике на карточку ситуации уходит сетевой запрос, по завершении которого на экране появляется сгенерированная отмазка и корректный скор.

### Шаг 4.2: Контейнеризация и финальные тесты (Роль: A-DQ)

* **Описание задачи:** Написать `Dockerfile` для бэкенда, настроить раздачу статического фронтенда и проверить устойчивость к Prompt Injection.
* **Входной контекст:** Весь готовый проект на текущий момент.
* **Действие:**
1. Создать `backend/Dockerfile` на базе официального легкого образа Python.
2. Настроить FastAPI на раздачу папки `../frontend/` как статических файлов (через `StaticFiles(directory="../frontend")`), чтобы сервис работал на одном порту.
3. Написать интеграционный тест на Python (используя `pytest` и `TestClient`), проверяющий, что отправка валидного Enum возвращает 200, а отправка произвольного текста вместо ID физически невозможна из-за валидации типов.


* **Критерии приемки:** Сборка Docker-контейнера проходит успешно. Контейнер запускается, сервис полностью работоспособен по адресу `http://localhost:8000`. Все автоматические тесты пройдены.
