## 👥 1. Матрица распределения ролей и контекстные ограничения

Каждый агент должен работать **строго** в рамках своего контекстного окна и файлового контура. Агентам запрещено модифицировать файлы, не входящие в их зону ответственности.

| Роль агента | Идентификатор | Зона ответственности (Файлы) | Ограничения доступа / Security Boundary |
| :--- | :--- | :--- | :--- |
| **Agent Backend** | `A-BE` | `/backend/app/` | Нет доступа к `/frontend/`. Запрещен прямой ввод текста в промпты LLM. Валидация строго через Enum. |
| **Agent Frontend** | `A-FE` | `/frontend/` | Нет доступа к `/backend/`. Запрещено придумывать дизайн-систему «из головы» без вызова MCP `google-stitch`. |
| **Agent DevOps/QA**| `A-DQ` | `/backend/Dockerfile`, `.env.example`, тесты | Доступ ко всему репозиторию в режиме Read-Only, Write-Only для инфраструктурных файлов. |

## 🤖 2. Инструкции для ИИ-Агентов

### Инструкция для Agent Backend (`A-BE`)
1. **Контекст:** Ты реализуешь Фазу 2 из `PLAN.md`. Твой главный ориентир — Секция 3 в `SPEC.md`.
2. **Алгоритм действий:**
   * Создай Pydantic-модели на основе контракта данных.
   * Настрой безопасную подстановку через `SITUATION_MAP` для защиты от **Prompt Injection**. При попытке передать невалидный ID, FastAPI должен автоматически отдавать `422 Unprocessable Entity`.
   * При интеграции с Gemini API передавай `response_schema` в `GenerateContentConfig` для обеспечения жесткого детерминизма JSON.
3. **Запреты:** Никогда не используй f-строки для прямой склейки пользовательского ввода с промптом к Gemini.

### Инструкция для Agent Frontend (`A-FE`)
1. **Контекст:** Ты реализуешь Фазу 3 и Шаг 4.1 из `PLAN.md`. Твой ориентир — Секция 4 в `SPEC.md`.
2. **Алгоритм действий:**
   * Сверстай интерфейс согласно дизайн-токенам.
   * Напиши JS-клиент для отправки POST-запросов на бэкенд.
   * **Критически важно:** При проектировании или изменении интерфейса ты обязан вызывать инструменты MCP-сервера `google-stitch`.
3. **Запреты:** Запрещено использовать сторонние CSS-фреймворки, кроме Tailwind CSS.

### Инструкция для Agent DevOps/QA (`A-DQ`)
1. **Контекст:** Ты реализуешь Фазу 1 и Шаг 4.2 из `PLAN.md`.
2. **Алгоритм действий:**
   * Подготовь окружение, зафиксируй версии зависимостей.
   * Напиши интеграционные тесты для проверки иммунитета к инъекциям (валидация типов).
   * Собери легковесный Docker-контейнер.

## 🔌 3. Интеграция с MCP Google Stitch (Инструкция для фронтенд-агента)

Агент `A-FE` должен использовать утилиты MCP-сервера `google-stitch` в следующих точках разработки:

### Точка вызова 1: Генерация палитры и тем оформления
* **Когда вызывать:** Перед началом верстки Шага 3.1 (`PLAN.md`), чтобы убедиться в соответствии контрастности и ироничного стиля.
* **Инструмент:** `google-stitch::generate_palette` или `google-stitch::get_tailwind_tokens`.
* **Промпт для вызова:** `"Сгенерируй палитру Tailwind для ироничного веб-сервиса в стиле корпоративного минимализма на основе Slate и Indigo"`.

### Точка вызова 2: Дизайн UI-компонентов
* **Когда вызывать:** Во время реализации Шага 3.2 при верстке виджета «Рейтинг правдоподобности» и карточек ситуаций.
* **Инструмент:** `google-stitch::get_component_layout`.
* **Промпт для вызова:** `"Выдай оптимальную разметку Tailwind для карточки выбора (Select Card) с состояниями hover и focus, а также для индикатора выполнения (progress badge)"`.

## 🔄 4. Синхронизация и моменты передачи контроля

Разработка ведется последовательно, переходя от агента к агенту по цепочке:

1.  `A-DQ` инициализирует репозиторий (Фаза 1) ➡️ передает управление `A-BE`.
2.  `A-BE` создает эндпоинты и логику Gemini (Фаза 2) ➡️ фиксирует контракты данных ➡️ передает управление `A-FE`.
3.  `A-FE` вызывает `google-stitch` ➡️ верстает интерфейс и связывает его с API (Фаза 3, Шаг 4.1) ➡️ передает управление `A-DQ`.
4.  `A-DQ` проводит финальное тестирование безопасности и контейнеризацию (Шаг 4.2) ➡️ Проект завершен.
